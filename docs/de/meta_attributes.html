    Allgemein¶

    «Einem Metaelement können kompositorisch Metaattribute zugeordnet sein (Klasse “MetaAttribut”). Metaattribute sind weder durch die Sprache INTERLIS noch durch das Metamodell genauer definiert. Sie dienen insbesondere dazu, dass Informationen, die über INTERLIS hinausgehen, als Bestandteil der Modelldaten beigefügt werden können. Metaattribute haben einen Namen (Attribut “Name”), der innerhalb der Metaattribute desselben Metaelementes eindeutig sein muss, und den beschreibenden Wert (Attribut “Value”).»

    Quelle: STAN_d_DEF_2011-06-22_eCH-0117 Meta-Attribute für INTERLIS-Modelle.pdf von eCH www.interlis.ch

    Metaattributes in Interlis Files¶

    Kommentare vs. Attribute¶

    Ein Interlis Kommentar startet mit !! und endet mit einem Linienende. Ein Metaattribut startet ebenfalls mit !! allerdings gefolgt von einem @:

    Syntax¶

    Nach dem Start von !!@ folgt der Attributname, dann ein Gleichheitszeichen = und dann der Attributwert.

    !!@<name>=<value>

    Gefolgt vom referenzierten Element (MODEL, TOPIC, CLASS etc.)

    Beispiel

    furtherInformation referenziert auf ExceptionalLoadsCatalogues_V1, topicInformation auf TypeOfRouteCatalogue und dispExpression auf TypeOfRoute.

    Für komplexere Anwendungsfälle siehe die Spezifikation vom Verein eCH

    Metaattribute in der Datenbank¶

    Das im Projektgenerator integrierte ili2db speichert die Metaatribute beim Datenimport vom Interlisfile in die Tabelle t_ili2db_meta_attrs:

    ilielement attr_name attr_value
    ExceptionalLoadsCatalogues_V1 furtherInformation https://www.astra.admin.ch/
    ExceptionalLoadsCatalogues_V1.TypeOfRouteCatalogue topicInformation Route-Type
    ExceptionalLoadsCatalogues_V1.TypeOfRouteCatalogue.TypeOfRoute dispExpression CONCAT(type, ‘ sometext’)

    Spezifische Projektgenerator-Metaattribute¶

    Einige nicht-standard Metaattibute werden vom Projektgenerator als Eigenschaften im QGIS-Projekt interpretiert.

    Liste spezifischer Attribute¶

    dispExpression wird als Anzeigeausdruck für einen Layer benutzt. Der Anzeigeausdruck definiert den Namen, der verwendet wird, wann immer ein Feature mit Text dargestellt werden muss. Beispielsweise in der Auswahlliste für das Beziehungsreferenzwidget auf Attributformularen.

    |relation_reference|

    Extra Model Information File¶

    Die Metaattribute können anstatt direkt im Interlis File, in diesen externen Files gespeichert sein.

    Diese Files sind in TOML geschrieben haben die Dateierweiterung .toml

    Extra Model Information Dateien können in den “Erweiterten Optionen” ausgewählt werden. Diese Konfiguration wird pro Model abgespeichert, was bedeutet, dass wenn man das Model später erneut auswählt, die Datei noch immer verknüpft ist. Diese Information wird auch im Hauptfenster des Projektgenerators angezeigt.

    Im Hintergrund schreibt ili2pg das Attribute von der Extra Model Information Datei in den PostGIS- oder GeoPackage-Speicher. Von dort kann es der Projektgenerater beim Erstellen des QGIS-Projektes verwenden.

    Beispiel

    ExceptionalLoadsRoute.toml¶
    ["ExceptionalLoadsRoute.TypeOfRouteCatalogue.TypeOfRoute"]
dispExpression="type"

    Komplexes Beispiel

    ExceptionalLoadsRoute.toml¶
    ["ExceptionalLoadsCatalogues_V1.TypeOfRouteCatalogue.TypeOfRoute"]
dispExpression="CONCAT(type, ' sometext')"

    Die Schlüssel, die für die TOML Sektionen verwendet werden sind “vollständig qualifizierte Interlisnamen” der Objekte. Im Beispiel oben ist dies ["ExceptionalLoadsCatalogues_V1.TypeOfRouteCatalogue.TypeOfRoute"]. Eine Liste von verfügbaren Namen kann in der Datenbanktabelle t_ili2db_classname gefunden werden, nachdem ein Schemaimport gemacht wurde.

    Ili Beispiel

    Obiges Beispiel kann wie folgt in eine ili-Datei geschrieben werden. Wenn sich diese Konfiguration in der ini-Datei befindet, wird das ili-Metaattribut nicht mehr benötigt. Hier steht es nur als Hinweis.

    ExceptionalLoadsRoute.ili¶
    !!@dispExpression="type"
CLASS TypeOfRoute=
  type: MANDATORY TEXT*25;
END TypeOfRoute;
